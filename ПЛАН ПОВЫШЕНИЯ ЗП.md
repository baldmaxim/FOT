# Заявка на повышение оклада + Динамическая система ролей

## Контекст

Две связанные задачи:
1. **Система ролей** — переход с hardcoded ENUM на динамическую таблицу `system_roles` с управлением через админ-панель. Необходимо для заявок на повышение, где нужны роли HR, Finance, C&B.
2. **Заявка на повышение оклада** — многоблочная форма с 3-этапным согласованием (руководитель → HR → финансы).

**Что уже есть в БД:** `current_salary`, `hire_date`, `position_name`, `department`, `work_object`, `supervisor_id`, таблица `employee_history` (история зарплат с `effective_date`), таблица `salary_history`.
**Чего нет:** `bonus_scheme`, `fot_profile` — будут `null` / "Не указано".

# ЧАСТЬ 1. Динамическая система ролей

## 1.1 Миграция БД

Файл: `docs/migrations/003_dynamic_roles.sql`

### Таблица `system_roles`

| Колонка | Тип | Описание |
|---------|-----|----------|
| id | UUID PK DEFAULT gen_random_uuid() | |
| code | TEXT UNIQUE NOT NULL | Код роли (worker, header, hr, finance...) |
| name | TEXT NOT NULL | Отображаемое имя (Сотрудник, Руководитель...) |
| hierarchy_level | INTEGER NOT NULL DEFAULT 0 | Уровень иерархии для `canAccess` (worker=1, header=2, hr=3, admin=4, super_admin=5) |
| is_system | BOOLEAN DEFAULT false | Защита от удаления (true для 5 начальных) |
| is_active | BOOLEAN DEFAULT true | Можно деактивировать без удаления |
| created_at | TIMESTAMPTZ DEFAULT NOW() | |

Начальные данные (INSERT):
```
(worker, Сотрудник, 1, true)
(header, Руководитель, 2, true)
(hr, Отдел кадров, 3, true)
(admin, Администратор, 4, true)
(super_admin, Супер-админ, 5, true)
```

### Таблица `role_page_access`

| Колонка | Тип | Описание |
|---------|-----|----------|
| id | UUID PK DEFAULT gen_random_uuid() | |
| role_code | TEXT NOT NULL → system_roles(code) ON DELETE CASCADE | |
| page_path | TEXT NOT NULL | Путь страницы (/employee, /dashboard, /admin/users...) |
| can_view | BOOLEAN DEFAULT true | Доступ на просмотр |
| can_edit | BOOLEAN DEFAULT false | Доступ на редактирование |
| UNIQUE(role_code, page_path) | | |

Начальные данные — заполнить из текущей конфигурации маршрутов:
- worker: `/employee/*`
- header: `/employee/*`, `/dashboard`, `/my-employees`, `/leave-requests`, `/timesheet`, `/profile`
- hr: всё header + `/timesheet-review`
- admin: всё hr + `/tender`, `/skud-raw`, `/skud-db`, `/discipline`
- super_admin: всё + `/admin/*`, `/skud-settings`

### Миграция данных

```sql
-- 1. Меняем тип колонки с ENUM на TEXT
ALTER TABLE user_profiles ALTER COLUMN position_type TYPE TEXT;
-- 2. Удаляем старый ENUM тип
DROP TYPE IF EXISTS employee_position_type;
-- 3. Добавляем FK constraint
ALTER TABLE user_profiles ADD CONSTRAINT fk_position_type
  FOREIGN KEY (position_type) REFERENCES system_roles(code);
```

Существующие пользователи автоматически сохранят свои значения — коды совпадают.

## 2.2 Стейт-машина

```
draft ──submit──► supervisor_review
                       │
              reject → rejected
              approve ──► hr_review
                              │
                     reject → rejected
                     approve ──► finance_review
                                      │
                             reject → rejected
                             approve ──► approved

draft / supervisor_review ──cancel──► cancelled
```

| Текущий статус | Действие | Кто может | Следующий |
|---|---|---|---|
| draft | submit | автор | supervisor_review |
| supervisor_review | approve | руководитель (supervisor_id ИЛИ header отдела) | hr_review |
| supervisor_review | reject | руководитель | rejected |
| hr_review | approve | роль с правом hr_review (hr/admin/super_admin) | finance_review |
| hr_review | reject | то же | rejected |
| finance_review | approve | роль с правом finance_review (finance/admin/super_admin) | approved |
| finance_review | reject | то же | rejected |
| draft, supervisor_review | cancel | автор | cancelled |
