# Yandex-target патчи для исторических миграций

## Зачем эта папка существует

Файлы в `docs/migrations/` — это **исторические артефакты**, применённые
на боевом Supabase. Их менять нельзя:

- они уже применены ручным `psql` на проде (см. CLAUDE.md: «авто-миграций
  нет, применяются вручную»);
- любая правка после применения создаёт расхождение между «что в репо»
  и «что в БД»;
- даже если в проекте нет checksum-based migration runner, для команды
  важно, чтобы git-blame по `docs/migrations/*.sql` отражал реальную
  историю изменений в боевой БД, а не пост-фактум «исправления».

При этом часть исторических миграций содержит конструкции, которые не
работают на Yandex Managed PostgreSQL (Supabase-only схемы, `NOTIFY pgrst`,
GRANT для Supabase-ролей и т. п.) — их **нужно учитывать** при заливке
схемы в Yandex.

## Как обрабатываются Yandex-несовместимости

Есть два независимых пути, и они **не конфликтуют**:

### 1. Schema-only dump + `prepare-yandex-schema.mjs` (рекомендуется)

При штатной миграции данных оператор:

1. снимает `pg_dump --schema-only` с боевой Supabase
   → `.migration/supabase_schema.sql`;
2. прогоняет через `scripts/yandex-migration/prepare-yandex-schema.mjs`
   → получает `.migration/yandex_schema{,_pre_data,_post_data}.sql`;
3. применяет на Yandex.

`prepare-yandex-schema.mjs` **автоматически стрипает** все Supabase-only
конструкции из дампа: `INSERT INTO storage.buckets`, `NOTIFY pgrst`,
`GRANT … TO anon/authenticated/service_role`, объекты в схемах
`auth/storage/realtime/…`, `CREATE EXTENSION`, RLS. Исходные файлы в
`docs/migrations/` при этом **никак не задействованы** — оператор
работает с дампом готовой схемы из живой Supabase.

Этот путь — основной. В нём патчи из этой папки **не применяются**.

### 2. Ручной прогон исторических миграций на Yandex с нуля (редкий случай)

Если оператор по какой-то причине заливает схему не из дампа, а
последовательно прогоняя `001…086` через `psql` на свежем Yandex-кластере,
часть миграций упадёт. Тогда **до** прогона надо применять патчи из этой
папки — они описывают, **что именно** в конкретной исторической миграции
нужно пропустить/изменить, чтобы она прошла на Yandex.

Патчи здесь — это **markdown-инструкции**, а не альтернативный SQL.
Оператор сам решает: пропустить блок, поменять команду или сгенерировать
свой адаптированный файл. Так мы не плодим SQL-форки исторических
миграций.

## Правила использования

- **Не редактируйте `docs/migrations/*.sql`** ни для исправления Supabase-only
  конструкций под Yandex, ни для уточнения комментариев. Если миграция
  уже применена в проде — она замёрзла.
- **Документируйте Yandex-расхождения тут**, в `docs/yandex-postgres-migration/patches/`.
- Если нашли некорректный комментарий в исторической миграции — добавьте
  заметку-патч `<NNN>_<name>.md` с заголовком `Comment correction` вместо
  правки самого `.sql`.
- В именах патчей сохраняйте префикс соответствующей миграции:
  `026_skud_object_maps.md`, `060_data_api.md` и т. п.
